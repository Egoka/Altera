# Структура GraphQL запросов

Этот документ описывает организацию GraphQL запросов в проекте Altera.

## 📁 Расположение файлов

Все GraphQL запросы находятся в директории `/web/app/query/` и организованы по функциональным областям.

## 📋 Структура директорий

```
web/app/query/
├── index.ts           # Центральный файл экспорта всех запросов
├── types.ts           # TypeScript типы для GraphQL ответов
├── pages/             # Запросы для публичных страниц
│   ├── home.ts          # Главная страница
│   ├── articles/
│   │   └── detail.ts    # Страница статьи
│   ├── authors/
│   │   └── detail.ts    # Страница автора
│   ├── tags/
│   │   └── detail.ts    # Страница тега
│   └── types/
│       └── detail.ts    # Страница типа контента
├── me/                # Личный кабинет пользователя
│   ├── profile.ts       # Профиль пользователя
│   └── articles.ts      # Управление своими статьями
├── admin/             # Административная панель
│   ├── articles.ts      # Управление всеми статьями
│   ├── tags.ts          # Управление тегами
│   └── types.ts         # Управление типами контента
└── common/            # Общие запросы
    ├── navigation.ts    # Навигация и меню
    ├── search.ts        # Поиск
    └── auth.ts          # Аутентификация
```
---
## 📄 Описание файлов

### 🏠 Главная страница (`pages/home`)

Для главной страницы доступны как комплексный запрос, так и отдельные запросы для каждой секции. Рекомендуется использовать комплексный запрос для минимизации количества сетевых запросов.

**Запросы:**
- **GET_HOME_PAGE_ARTICLES**: Комплексный запрос для главной страницы, включающий все три секции, описанные ниже.
- **GET_FEATURED_ARTICLES**: Запрос для получения главных материалов (5 статей).
- **GET_LATEST_ARTICLES**: Запрос для получения последних статей (20 статей, исключая главные).
- **GET_POPULAR_ARTICLES**: Запрос для получения популярных статей за неделю (10 статей).

**Поля в комплексном запросе:**
- `featuredArticles` - Главные материалы (5 статей)
- `latestArticles` - Последние статьи (20 статей, исключая главные)
- `popularArticles` - Популярные за неделю (10 статей)
---
### 📰 Страница статьи (`pages/articles/detail`)

Для страницы статьи доступны как комплексные запросы, так и отдельные запросы для каждой части данных. Рекомендуется использовать комплексные запросы для минимизации количества сетевых запросов.

**Комплексные запросы:**
- **GET_ARTICLE_PAGE**: Комплексный запрос для страницы статьи, включающий все части, описанные ниже.
- **GET_ARTICLE_DETAIL**: Альтернативный комплексный запрос, возвращающий структурированный объект с полной информацией.

**Отдельные запросы:**
- **GET_ARTICLE**: Запрос для получения полной информации о статье (включая body, автора, тип контента, теги)
- **GET_RECOMMENDED_ARTICLES**: Запрос для получения рекомендуемых статей (похожие по тегам)
- **GET_RELATED_ARTICLES**: Запрос для получения связанных статей (по автору и типу контента)
- **GET_ARTICLE_STATS**: Запрос для получения статистики статьи (время чтения, количество слов)

**Поля в комплексном запросе:**
- Полная информация о статье (заголовок, содержание, автор, тип контента, теги)
- Рекомендуемые статьи (5 статей, похожих по тегам)
- Связанные статьи (10 статей от того же автора или того же типа контента)
- Статистика статьи (время чтения, количество слов, просмотры, шары)

**Алгоритмы рекомендаций:**
- **Рекомендуемые статьи**: Находятся по общим тегам с текущей статьей
- **Связанные статьи**: Находятся по автору или типу контента
- **Статистика**: Рассчитывается на основе количества слов (примерно 200 слов в минуту)

### 👤 Страница автора (`pages/authors/detail`)

Для страницы автора доступны как комплексный запрос `GET_AUTHOR_PAGE`, так и отдельные запросы для каждой части данных. Рекомендуется использовать комплексный запрос для минимизации количества сетевых запросов.

**Запросы:**
- **GET_AUTHOR_PAGE**: Комплексный запрос, включающий три части, описанные ниже.
- **GET_AUTHOR**: Запрос для получения публичной информации об авторе (имя, био, фото, соцсети).
- **GET_AUTHOR_ARTICLES**: Запрос для получения списка опубликованных статей конкретного автора с пагинацией.
- **GET_AUTHOR_STATS**: Запрос для получения статистики (общее число статей, самые используемые теги и т.д.).
---
### 🏷️ Страница тега (`pages/tags/detail`)

Для страницы тега доступны как комплексный запрос `GET_TAG_PAGE`, так и отдельные запросы для каждой части данных. Рекомендуется использовать комплексный запрос для минимизации количества сетевых запросов.

**Запросы:**
- **GET_TAG_PAGE**: Комплексный запрос для страницы тега, включающий три части, описанные ниже.
- **GET_TAG**: Запрос для получения публичной информации о теге (название, описание, дата создания).
- **GET_TAG_ARTICLES**: Запрос для получения списка статей с этим тегом с пагинацией.
- **GET_TAG_STATS**: Запрос для получения статистики тега (общее число статей, статьи за месяц, среднее время чтения, популярные авторы).

**Поля в комплексном запросе:**
- Информация о теге (название, описание, дата создания)
- Список статей с этим тегом (с пагинацией)
- Статистика тега (количество статей, популярные авторы, среднее время чтения)
---
### 📚 Страница типа контента (`pages/types/detail`)

Для страницы типа контента доступны как комплексный запрос `GET_CONTENT_TYPE_PAGE`, так и отдельные запросы для каждой части данных. Рекомендуется использовать комплексный запрос для минимизации количества сетевых запросов.

**Запросы:**
- **GET_CONTENT_TYPE_PAGE**: Комплексный запрос для страницы типа контента, включающий три части, описанные ниже.
- **GET_CONTENT_TYPE**: Запрос для получения публичной информации о типе контента (название, описание, порядок, статус).
- **GET_CONTENT_TYPE_ARTICLES**: Запрос для получения списка статей этого типа с пагинацией.
- **GET_CONTENT_TYPE_STATS**: Запрос для получения статистики типа контента (общее число статей, статьи за месяц, среднее время чтения, популярные теги, топ авторы).

**Поля в комплексном запросе:**
- Информация о типе контента (название, описание, порядок, статус)
- Список статей этого типа (с пагинацией)
- Статистика типа контента (количество статей, популярные теги, топ авторы, среднее время чтения)
---
### 👤 Личный кабинет - Профиль (`me/profile`)

Для профиля пользователя доступны запросы для получения персональной информации и статистики по статьям. Все запросы требуют аутентификации.

**Запросы:**
- **GET_MY_PROFILE**: Получение профиля текущего пользователя (персональная информация, роль, социальные ссылки)
- **GET_MY_ARTICLES_STATS**: Статистика по статьям пользователя (общее количество, по статусам: опубликовано, черновики, на рецензии, архивные)

**Поля в профиле (запрос `me`):**
- Персональная информация (имя, email, био, фото, роль)
- Социальные ссылки (JSON)
- Даты создания и обновления профиля

**Поля в статистике (запрос `myArticlesStats`):**
- `total` - Общее количество статей
- `published` - Количество опубликованных статей
- `draft` - Количество черновиков
- `review` - Количество статей на рецензии
- `archived` - Количество архивных статей
---
### 📝 Личный кабинет - Статьи (`me/articles`)

Для управления статьями в личном кабинете доступны запросы для получения данных и мутации для CRUD операций. Все операции требуют аутентификации и применяются только к статьям текущего пользователя.

**Запросы:**
- **GET_MY_ARTICLES**: Список моих статей с фильтрацией по статусу и пагинацией
- **GET_ARTICLE_FOR_EDIT**: Получение полной информации о статье для редактирования (включая все поля и связи)
- **GET_MY_ARTICLES_STATS**: Статистика по моим статьям (общее количество, по статусам, по месяцам)

**Мутации для создания и редактирования:**
- **CREATE_ARTICLE**: Создание новой статьи (всегда создается со статусом `draft`)
- **UPDATE_ARTICLE**: Обновление существующей статьи (автор может изменять только свои статьи)

**Мутации для управления статусом:**
- **REQUEST_REVIEW**: Запрос на рецензирование статьи (перевод из `draft` в `review`)
- **REVERT_TO_DRAFT**: Возврат статьи в черновик (перевод из `review` в `draft`)
- **ARCHIVE_ARTICLE**: Архивирование статьи (мягкое удаление, перевод в `archived`)

**Мутации для удаления:**
- **DELETE_ARTICLE**: Физическое удаление статьи (только для черновиков и архивных статей)

**Правила доступа:**
- Автор может создавать, редактировать и архивировать только свои статьи
- Автор может запрашивать рецензирование только своих черновиков
- Автор может возвращать в черновик только свои статьи на рецензии
- Физическое удаление доступно только для черновиков и архивных статей
- Изменение статуса на `published` доступно только администраторам

## 🎯 Использование

### Импорт запросов

```typescript
// Импорт отдельных запросов
import { GET_HOME_PAGE_ARTICLES } from '~/query/pages/home'
import { GET_ARTICLE } from '~/query/pages/articles/detail'

// Импорт всех запросов через index
import { 
  GET_HOME_PAGE_ARTICLES, 
  GET_ARTICLE,
  SEARCH_ARTICLES 
} from '~/query'
```

### Пример использования в компоненте Nuxt

```vue
<script setup lang="ts">
import { GET_HOME_PAGE_ARTICLES } from '~/query'
import type { ArticleListItem } from '~/query/types'

const { data } = await $fetch('/api/graphql', {
  method: 'POST',
  body: {
    query: GET_HOME_PAGE_ARTICLES
  }
})

const featuredArticles = ref<ArticleListItem[]>(data.featuredArticles)
</script>
```

## 📊 TypeScript типы

Файл `types.ts` содержит интерфейсы для:

- **ArticleListItem** - Базовая информация о статье для списков
- **ArticleDetail** - Полная информация о статье
- **User** - Информация о пользователе
- **SectionTag** - Информация о теге
- **ContentType** - Информация о типе контента
- **Pagination** - Общий интерфейс пагинации
- **ArticlesResponse** - Ответ со списком статей
- **SearchSuggestionsResponse** - Ответ с предложениями поиска
- **ArticleStatsResponse** - Статистика по статьям
- **AuthorStatsResponse** - Статистика автора

## 🚀 Особенности реализации

1. **Модульная структура** - Запросы разделены по функциональным областям
2. **Простой экспорт** - Запросы экспортируются как строки, без зависимости от graphql-tag
3. **Типизация** - Полная типизация для TypeScript
4. **Пагинация** - Встроенная поддержка пагинации для всех списков
5. **Переиспользование** - Общие фрагменты могут быть вынесены в отдельные константы

## 📝 Соглашения по именованию

- **GET_** - префикс для запросов (queries)
- **CREATE_**, **UPDATE_**, **DELETE_** - префиксы для мутаций
- **BULK_** - префикс для массовых операций
- PascalCase для констант с запросами
- camelCase для полей в запросах

## 🔄 Обновление запросов

При добавлении новых запросов:

1. Создайте файл в соответствующей директории
2. Экспортируйте запрос как константу
3. Добавьте экспорт в `index.ts`
4. При необходимости добавьте типы в `types.ts`
5. Обновите эту документацию
